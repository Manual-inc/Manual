# Adaptive Compute

## 정의

Adaptive Compute는 워크플로우를 지속적으로 개선해 비용을 최소화하는 운영 원칙이다. Manual에서는 각 과제를 결정성, 난이도, 실패 비용, 필요한 추론 깊이에 따라 다른 실행 수단으로 배치하고, 실행 결과와 비용 데이터를 바탕으로 그 배치를 계속 개선하는 것을 뜻한다.

핵심은 모든 일을 상위 LLM에 맡기지 않는 것이다. 결정적으로 처리 가능한 과제는 스크립트나 규칙 기반 도구로 처리하고, 낮은 위험의 비결정적 과제는 로컬 LLM 또는 비용이 낮은 모델에 맡기며, 가장 핵심적이고 고심이 필요한 비결정적 과제에만 상위 LLM을 사용한다.

## 용어

- 결정적 과제: 같은 입력에 대해 같은 출력이 기대되는 작업. 예: 파일 목록 수집, 의존성 그래프 생성, 테스트 실행, 정적 분석, 포맷 검사, JSON schema 검증.
- 비결정적 과제: 해석, 판단, 요약, 우선순위화, 설계 선택처럼 모델의 추론과 맥락 이해가 필요한 작업.
- 저비용 모델: 로컬 LLM, 작은 원격 모델, 또는 낮은 가격의 provider 모델. 높은 창의성이나 깊은 판단이 필요하지 않은 분류, 초안, 요약, 라우팅에 적합하다.
- 상위 LLM: 비용은 높지만 복잡한 추론, 코드 이해, 애매한 요구사항 해석, 위험한 의사결정에 강한 모델.
- 라우팅: 노드 또는 하위 과제를 어떤 실행기와 모델에 맡길지 결정하는 과정.
- 승격: 저비용 실행으로 충분하지 않거나 불확실성이 높을 때 더 강한 모델 또는 사람 검토로 넘기는 과정.
- 비용 회귀: 워크플로우 변경 후 동일하거나 유사한 작업에서 토큰, 시간, 실패율, 재시도 비용이 증가하는 현상.

## 계층형 처리 전략

Adaptive Compute는 과제를 아래 순서로 낮은 비용 계층부터 검토한다.

1. 스크립트 또는 정적 도구로 해결할 수 있는가?
2. 규칙, schema, 캐시, 이전 실행 결과로 해결할 수 있는가?
3. 로컬 LLM 또는 저비용 모델이 충분히 처리할 수 있는가?
4. 상위 LLM이 필요한 수준의 추론, 설계, 검증, 불확실성 처리가 있는가?
5. 모델 판단만으로는 위험하므로 사람 확인이 필요한가?

이 순서는 고정 실행 순서라기보다 설계 원칙이다. 예를 들어 상위 LLM이 먼저 전체 계획을 만들더라도, 실제 실행 단계는 스크립트와 저비용 모델로 분해해 비용을 낮출 수 있다.

## 구체적인 방법

### 1. 노드별 과제 분류

워크플로우의 각 노드를 다음 속성으로 분류한다.

- 결정성: deterministic, mostly-deterministic, non-deterministic.
- 요구 추론 깊이: low, medium, high.
- 실패 비용: low, medium, high.
- 검증 가능성: 자동 검증 가능, 샘플 검증 가능, 사람 검토 필요.
- 입력 크기: 작음, 중간, 큼.
- 재사용 가능성: 캐시 가능, 부분 캐시 가능, 매번 새로 계산 필요.

이 분류는 [[node-storybook|노드 Storybook]]과 잘 맞는다. 노드별 입력/출력 예시를 쌓으면 어떤 노드가 스크립트로 대체 가능한지, 어떤 노드는 저비용 모델로 충분한지 실험할 수 있다.

### 2. 결정적 단계의 스크립트화

LLM이 반복적으로 수행하지만 실제로는 결정적인 작업을 찾아 스크립트 노드로 바꾼다.

- repository 탐색, 파일 검색, 라인 수 계산, dependency 추출.
- 테스트 실행, 빌드, lint, format, typecheck.
- JSON/YAML/TOML 파싱과 schema 검증.
- 로그 필터링, diff 생성, 단순 리포트 작성.
- 문서 frontmatter 검사, 링크 유효성 검사, orphan page 탐지.

스크립트화는 토큰을 줄이는 동시에 재현성을 높인다. LLM은 계산 자체가 아니라 결과 해석과 다음 행동 결정에 집중한다.

### 3. 에이전트 실행 로그 기반 노드 분리

에이전트 노드가 실제로 어떤 절차로 작업하는지 실행 로그를 분석해, 반복적으로 관찰되는 결정적 작업 과정을 별도 노드로 분리할 수 있어야 한다. 처음에는 하나의 에이전트 노드가 "상황 파악, 파일 검색, 테스트 실행, 결과 요약, 수정 전략 선택"을 모두 수행하더라도, 여러 실행에서 같은 deterministic sub-step이 계속 나타나면 그 부분은 스크립트 노드 또는 규칙 기반 노드로 승격한다.

예를 들어 에이전트가 매번 `rg`로 특정 파일군을 찾고, `cargo test`를 실행하고, 실패 로그에서 같은 패턴을 추출한다면 이 과정은 독립 노드가 될 수 있다. 이후 에이전트는 이미 구조화된 결과를 입력으로 받아 더 높은 수준의 판단만 수행한다.

이 방식은 워크플로우를 처음부터 완벽하게 설계하지 않아도 되게 한다. Manual은 실행 로그를 통해 에이전트의 실제 작업 습관을 관찰하고, 반복되는 결정적 구간을 후보로 제안하며, 사용자가 승인하면 노드를 분리해 워크플로우를 더 저렴하고 재현 가능하게 만든다.

노드 분리 후보를 찾을 때 볼 수 있는 신호는 다음과 같다.

- 여러 run에서 같은 명령, 같은 파일 탐색, 같은 검증 절차가 반복된다.
- 에이전트가 수행한 단계의 출력 형식이 안정적이고 schema로 표현 가능하다.
- 해당 단계의 성공/실패를 자동으로 판정할 수 있다.
- 해당 단계가 상위 LLM의 판단보다 도구 실행과 파싱에 가깝다.
- 분리 후 다음 에이전트 노드에 전달할 요약 또는 artifact가 명확하다.

### 4. 저비용 모델 라우팅

비결정적이지만 위험이 낮은 작업은 저비용 모델로 보낸다.

- 긴 로그나 문서의 1차 요약.
- 이슈/파일/노드의 단순 분류.
- 후보 목록 생성.
- 중복 제거와 클러스터링 초안.
- 프롬프트 입력을 줄이기 위한 전처리 요약.

저비용 모델의 출력은 가능하면 schema로 제한하고, 상위 LLM에는 원문 전체가 아니라 검증된 요약, 후보, 불확실성 표시를 전달한다.

### 5. 상위 LLM 사용 지점 명시

상위 LLM은 다음처럼 고심이 필요한 병목에 집중시킨다.

- 요구사항이 모호하고 제품 판단이 필요한 설계.
- 여러 모듈에 걸친 코드 변경 계획.
- 실패 원인이 복합적인 디버깅.
- 사용자 의도와 트레이드오프를 함께 해석해야 하는 결정.
- 저비용 모델 결과가 서로 충돌하거나 신뢰도가 낮은 경우의 판정.

상위 LLM 노드는 왜 상위 모델이 필요한지 이유를 metadata로 남기는 것이 좋다. 나중에 비용 최적화 시 해당 이유가 여전히 유효한지 검토할 수 있다.

### 6. 관측 기반 개선 루프

Adaptive Compute는 일회성 설계가 아니라 반복 개선 루프다.

1. 실행한다.
2. [[token-cost-observability|토큰 비용 관측]]으로 노드별 토큰, 비용, 시간, 실패율을 기록한다.
3. 비용이 큰 노드와 실패가 잦은 노드를 찾는다.
4. 에이전트 실행 로그에서 반복되는 결정적 구간을 찾아 노드 분리 후보로 표시한다.
5. 스크립트화, 캐싱, 입력 축소, 저비용 모델 대체, 상위 LLM 승격 중 하나를 적용한다.
6. 변경 전후 비용과 성공률을 비교한다.
7. 개선이 확인되면 워크플로우 템플릿 또는 노드 정책으로 고정한다.

## 라우팅 정책 예시

```yaml
node_policy:
  collect_context:
    task_type: deterministic
    runner: script
    cache: true
  summarize_logs:
    task_type: non_deterministic
    reasoning_depth: low
    runner: local_llm
    output_schema: log_summary.v1
  choose_fix_strategy:
    task_type: non_deterministic
    reasoning_depth: high
    runner: premium_llm
    escalation_reason: "multi-file debugging and product tradeoff"
  verify_patch:
    task_type: mostly_deterministic
    runner: script
    commands: ["cargo test", "npm test"]
```

## Manual 제품에서의 의미

Adaptive Compute는 Manual의 핵심 차별점이 될 수 있다. Manual이 단순히 여러 에이전트를 실행하는 도구라면 비용은 사용자가 직접 감당해야 한다. 반대로 Manual이 실행 기록을 학습해 "이 단계는 스크립트로 충분하다", "이 요약은 로컬 모델로 낮춰도 된다", "이 판단은 상위 LLM을 유지해야 한다"를 제안하면, 워크플로우 자체가 시간이 지날수록 싸고 안정적으로 변한다.

이를 위해 Manual은 다음 기능을 장기적으로 지원할 수 있다.

- 노드별 runner/model 정책 설정.
- 실행 결과 기반 비용 최적화 제안.
- 상위 LLM 호출 사유와 승격 조건 기록.
- 에이전트 실행 로그에서 반복되는 결정적 작업 구간을 감지하고 노드 분리 후보로 제안.
- 저비용 모델 결과와 상위 LLM 결과의 품질 비교.
- 동일 워크플로우의 변경 전후 비용 회귀 감지.
- 캐시 가능한 deterministic 노드 자동 식별.
- "이 워크플로우를 Adaptive Compute 관점에서 개선해줘" 같은 분석 명령.

## 열린 질문

- 노드 metadata에 모델 라우팅 정책을 얼마나 명시적으로 저장할 것인가?
- 저비용 모델의 품질을 어떤 기준으로 자동 평가할 것인가?
- 비용 절감과 결과 품질 사이의 허용 가능한 tradeoff를 사용자가 어떻게 설정할 것인가?
- 상위 LLM 승격 조건을 수동 규칙으로 둘 것인가, 실행 기록 기반 추천으로 발전시킬 것인가?
- 로컬 LLM 사용 시 privacy, latency, hardware capacity를 어떤 정책으로 함께 고려할 것인가?
