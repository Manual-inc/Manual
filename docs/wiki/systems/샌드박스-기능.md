# 샌드박스 기능

샌드박스 기능은 에이전트와 스크립트를 사전에 정의된 실행 정책 안에서 실행하게 해 [[실행-안전성-부족]]을 줄이는 기능이다.

핵심 아이디어는 사용자가 매 커맨드마다 위험을 판단하는 대신, 작업 유형별로 허용된 파일 접근, 실행 권한, 네트워크 접근 범위를 미리 정의하는 것이다. 이후 워크플로우 노드에서 에이전트나 스크립트를 실행할 때 해당 샌드박스를 지정하면, 실행은 그 정책 안에서만 허용된다.

## Purpose

샌드박스 기능의 목적은 안정성과 사용성을 함께 높이는 것이다.

- 사용자는 반복적인 승인 요청을 줄일 수 있다.
- 복잡한 커맨드를 매번 해석하지 않아도 된다.
- 에이전트나 스크립트가 실수해도 허용되지 않은 파일, 명령, 네트워크 접근을 차단할 수 있다.
- 기업 환경에서는 작업별 실행 경계를 사전에 정의하고 감사 가능하게 관리할 수 있다.

## Policy Model

샌드박스 정책은 macOS Seatbelt 또는 `sandbox-exec` 스타일의 허용/거부 규칙처럼, 실행 가능한 범위를 명시적으로 정의한다.

정책은 다음 범주를 포함할 수 있다.

- 파일 읽기 허용 경로
- 파일 쓰기 허용 경로
- 실행 가능한 파일 또는 명령
- 실행 금지 파일 또는 명령
- 접근 가능한 네트워크 호스트
- 접근 금지 네트워크 호스트
- 외부 프로세스 실행 가능 여부
- 환경 변수 접근 범위
- 임시 디렉터리와 캐시 디렉터리 사용 범위

MVP에서는 모든 운영체제 기능을 완벽하게 감싸기보다, Manual 워크플로우가 요구하는 파일, 커맨드, 네트워크 경계를 명확히 표현하는 정책 모델부터 시작한다.

## Sandbox Types

Manual은 재사용 가능한 샌드박스 종류를 관리해야 한다. 사용자는 매번 정책을 새로 작성하지 않고, 작업 유형에 맞는 샌드박스를 선택할 수 있어야 한다.

예시 샌드박스는 다음과 같다.

- Read-Only Repo: 저장소 전체 읽기만 허용하고 쓰기와 외부 네트워크를 차단한다.
- Docs Writer: `docs/` 하위 쓰기와 문서 검사 실행만 허용한다.
- Test Runner: 테스트 실행과 빌드 캐시 쓰기만 허용한다.
- Network Research: 지정된 호스트에 대한 읽기 목적 네트워크 요청만 허용한다.
- Local Script: 특정 스크립트 실행과 제한된 입출력 경로만 허용한다.

샌드박스 종류 관리 기능은 정책 생성, 복제, 수정, 버전 관리, 사용 이력 확인을 포함할 수 있다.

## Workflow Integration

워크플로우가 여러 노드로 쪼개져 있을 때, 에이전트나 스크립트를 실행하는 노드는 샌드박스를 지정해야 한다.

예를 들어 다음과 같이 구성할 수 있다.

```text
Workflow Node
Name: Documentation Update
Executor: Agent
Sandbox: Docs Writer
Allowed writes: docs/wiki/**
Allowed commands: cargo run in docs/test
Network: denied
```

이 방식은 실행 안정성을 워크플로우 정의의 일부로 만든다. 실행자가 에이전트든 스크립트든, 노드에 지정된 샌드박스 정책을 벗어나면 실행이 차단된다.

제품 방향 회의에서 논의된 [[agent-sandboxing|에이전트 샌드박스]]는 이 기능의 아키텍처 관점이다. 파일, 디렉터리, 네트워크, host 접근을 제한하고, 코드 리뷰 노드에는 필요한 외부 정보원만 허용하는 식의 정책이 언급됐다.

## macOS GUI Surface

mac GUI는 샌드박스를 별도 패널에서 관리한다. 사용자는 재사용 가능한 샌드박스 종류를 만들고, 파일 읽기/쓰기 경로, 실행 허용/거부 명령, 네트워크 허용/거부 호스트, 환경 변수 범위, 임시/캐시 쓰기 범위를 편집할 수 있다.

노드 인스펙터는 선택된 워크플로우 노드의 샌드박스 지정 상태를 보여주고, 선택한 샌드박스를 노드의 `sandbox_policy.sandbox_id`로 저장한다. app-server는 실행 직전에 이 ID를 실제 샌드박스 정책으로 materialize한다. 정책 패널의 probe는 `sandbox.evaluate` 결과를 표시해 승인 없이 허용되는 접근과 즉시 차단되어야 하는 접근을 사용자가 확인하게 한다.

## Relationship to Optimization

샌드박스 기능은 [[매뉴얼-최적화-기능]]과도 연결된다. 최적화 기능이 반복 작업을 스크립트화하거나 전처리하라고 추천할 때, 그 스크립트는 적절한 샌드박스 안에서 실행되어야 한다.

이 연결은 중요한 제품 가치를 만든다. Manual은 단순히 "스크립트로 자동화하라"고 제안하는 것이 아니라, "이 범위 안에서만 안전하게 자동화하라"고 제안할 수 있다.

## Relationship to Self-Evolution

[[매뉴얼-자기진화-기능]]이 매뉴얼을 자동 또는 반자동으로 개선할 때도 샌드박스가 필요하다. 자기진화 과정에서 생성된 스크립트나 새 워크플로우 단계는 실행 범위가 명확히 제한되어야 한다.

낮은 위험도의 자동 개선이라도 파일 쓰기, 실행, 네트워크 접근을 포함한다면 샌드박스 정책을 요구해야 한다. 이렇게 해야 자기진화가 편리함을 높이면서도 실행 위험을 키우지 않는다.

## Product Appeal

샌드박스 기능은 사용자와 기업 모두에게 매력적인 안전 기능이 될 수 있다.

개인 사용자에게는 "매번 승인하지 않아도 안전한 범위 안에서 에이전트가 일한다"는 편의성을 준다. 기업 사용자에게는 "에이전트가 접근할 수 있는 파일, 명령, 네트워크가 정책으로 제한된다"는 도입 근거를 제공한다.

이 기능은 Manual의 포지션을 단순 AI 워크플로우 도구가 아니라, 안전하게 에이전트를 운영하는 실행 시스템으로 강화한다.

## Open Questions

- 샌드박스 정책은 어떤 형식으로 저장할 것인가?
- 네트워크 호스트 제한은 도메인, IP, 포트 중 어디까지 표현할 것인가?
- 정책 위반 시 에이전트에게 재시도 기회를 줄 것인가, 즉시 실패시킬 것인가?
- 샌드박스 종류는 프로젝트별로 관리할 것인가, 사용자 전역으로 관리할 것인가?
- 정책 변경은 어떤 승인 절차와 이력 관리를 거쳐야 하는가?
