# 테스트 규칙

테스트 규칙은 Manual 프로젝트의 문서, 행동 계약, 앱 구현, 모듈 내부 로직을 어떤 테스트 계층으로 검증할지 정리하는 최상위 규칙이다.

Manual은 [[매뉴얼-시스템]]의 신뢰성을 문서만으로 보장하지 않는다. 문서와 유스케이스가 코드와 연결되고, CI에서 실행 가능한 테스트로 확인되어야 한다.

## Test Layers

Manual의 테스트는 다음 계층으로 나눈다.

- [[문서-규칙-테스트]] — `docs/` 내부의 llm-wiki 문서 구조, 문서 링크, 유스케이스-위키 추적 링크를 검증한다.
- [[기능-계약-테스트]] — `docs/usecase/*.feature`에 정의된 사용자 행동 계약을 각 앱이나 엔드 구현이 충족하는지 검증한다.
- Module unit tests — 각 모듈 내부의 세부 로직, parser, storage, runner, policy evaluator 같은 단위 동작을 해당 모듈의 테스트 위치에서 검증한다.

## CI Gate

CI는 테스트 계층을 함께 실행해야 한다.

- CI는 `docs/test`의 [[문서-규칙-테스트]]를 실행한다.
- CI는 `manual-rs/crates/app-server/`, CLI, macOS 앱, Windows 앱, Android 앱 등 존재하는 각 구현 디렉터리의 [[기능-계약-테스트]]를 실행한다.
- CI는 각 모듈의 단위 테스트를 실행한다.
- 문서 규칙 테스트, feature contract test, 모듈 단위 테스트 중 필수 테스트가 실패하면 머지할 수 없어야 한다.
- CI 결과는 어떤 계층의 어떤 테스트가 실패했는지 추적 가능하게 출력해야 한다.

## Module Unit Tests

모듈 내부의 단위 테스트는 각 모듈에 둔다.

- 모듈 단위 테스트는 해당 모듈의 소스 디렉터리나 그 언어/플랫폼의 표준 테스트 위치에 둔다.
- 단위 테스트는 내부 알고리즘, 경계 조건, 오류 처리, 데이터 변환을 빠르게 검증한다.
- `.feature` 구현 테스트는 여러 모듈을 묶어 사용자 시나리오가 실제로 충족되는지 검증한다.
- 새 모듈을 추가할 때는 관련 위키 문서 링크 규칙을 지키고, 모듈 내부 테스트도 함께 고려한다. 관련 규칙은 [[문서-연결-코드]]를 참고한다.

## Relationship

이 문서는 테스트 계층의 상위 지도를 제공한다. 세부 규칙은 [[문서-규칙-테스트]]와 [[기능-계약-테스트]]에 나누어 둔다.
