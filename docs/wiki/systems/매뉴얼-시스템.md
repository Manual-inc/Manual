# Manual System

Manual System은 AI 에이전트를 실제 업무에 도입할 때 [[토큰-부족]]와 [[신뢰성-부족]]를 함께 줄이기 위한 전체 작업 시스템이다.

## Purpose

Manual System의 목적은 모델이 틀릴 수 있다는 사실을 전제로, 토큰 사용과 작업 신뢰성을 시스템 차원에서 관리하는 것이다. 이 시스템은 단순히 LLM에게 더 많은 컨텍스트를 주거나 더 좋은 모델을 쓰는 방식이 아니라, 실행 구조, 검증 절차, 문서화, 코드 추적성을 함께 설계한다.

## Scope

Manual System은 [[LLM-Harness]]를 포함할 수 있지만, 하네스 자체와 동일한 개념은 아니다.

[[LLM-Harness]]는 LLM을 다루기 위한 도구, 환경, 권한, 실행 제어, 컨텍스트 주입 같은 기반 계층을 뜻한다. Manual System은 그 기반을 활용해 실제 업무 결과의 신뢰성, 비용 효율성, 검증 가능성을 달성하려는 더 넓은 작업 체계다.

## Responsibilities

- 에이전트가 따라야 할 작업 절차를 명확히 한다.
- 중요한 단계가 생략되지 않도록 체크포인트를 둔다.
- 결과물을 검증 가능한 단위로 나눈다.
- 테스트, 리뷰, 로그, 산출물을 통해 오류 원인을 추적할 수 있게 한다.
- 필요한 경우 토큰 사용량과 모델 선택을 [[토큰-배분]] 원칙에 따라 제어한다.
- 에이전트와 스크립트 실행을 [[샌드박스-기능]]으로 제한해 파일, 명령, 네트워크 접근 범위를 관리한다.
- 코드가 관련 문서와 연결되도록 [[문서-연결-코드]] 규칙을 유지한다.

## MVP Surface

[[매뉴얼-최적화-기능]]은 Manual System의 초기 제품 표면이다. 이 기능은 AI 워크플로우의 토큰 사용량, 검증 상태, 소요 시간을 측정하고, 측정 결과를 바탕으로 비교, 진단, 추천을 제공한다.

[[매뉴얼-자기진화-기능]]은 최적화 결과를 매뉴얼 자체의 개선으로 연결하는 확장 기능이다. 사용자는 단일 에이전트 매뉴얼로 시작할 수 있고, Manual은 실행 데이터와 검증 결과를 바탕으로 전처리, 스크립트화, 검증 절차, 모델 선택을 점진적으로 개선한다.

[[샌드박스-기능]]은 에이전트와 스크립트가 안전한 실행 경계 안에서 동작하도록 보장하는 실행 안전성 기능이다. 워크플로우 노드가 커맨드나 에이전트를 실행할 때, 해당 노드는 어떤 샌드박스 정책을 사용할지 지정해야 한다.

## Relationship to Reliability

Manual System은 [[확률적-모델-리스크]]를 제거하지 않는다. 대신 모델의 확률적 오류 가능성을 인정하고, 절차적 제약과 검증 가능한 산출물로 [[신뢰성-부족]]를 줄인다. 또한 샌드박스 정책으로 실행 경계를 제한해 [[실행-안전성-부족]]을 줄인다.

## Open Design Questions

- 어떤 절차를 강제하고 어떤 절차를 권고로 둘 것인가?
- Manual System은 단일 에이전트를 감싸는 구조로 시작할 것인가, 아니면 여러 역할을 분리한 구조로 시작할 것인가?
- [[LLM-Harness]]가 담당할 실행 제어와 Manual System이 담당할 업무 검증의 경계를 어디에 둘 것인가?
- 신뢰성 향상은 어떤 지표로 측정할 것인가?
