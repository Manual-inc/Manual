# 매뉴얼 자기진화 기능

매뉴얼 자기진화 기능은 사용자가 처음 등록한 단일 에이전트 기반 매뉴얼이 [[매뉴얼-최적화-기능]]의 측정, 진단, 추천, 검증 결과를 바탕으로 점진적으로 더 효율적인 작업 시스템으로 변해가는 기능이다.

초기 상태의 매뉴얼은 Codex나 Claude Code 같은 일반적인 [[단일-에이전트-오케스트레이션]]에 가깝게 시작할 수 있다. 사용자는 하나의 에이전트에게 작업을 맡기고, Manual은 그 실행 결과를 관찰한다. 이후 반복 실행이 쌓이면 Manual은 어떤 단계가 반복되는지, 어디서 토큰이 낭비되는지, 어떤 검증이 부족한지, 어떤 전처리가 필요한지를 찾아 매뉴얼 자체를 개선한다.

## Product Appeal

이 기능의 매력은 Manual이 단순한 모니터링 도구에 머물지 않는다는 점이다. 사용자는 처음부터 완성된 워크플로우를 설계하지 않아도 된다. 일단 평소처럼 에이전트에게 일을 맡기면, Manual이 실행 데이터를 바탕으로 더 나은 매뉴얼을 제안하거나 자동으로 개선한다.

즉, Manual은 "AI 작업을 측정하는 도구"에서 "AI 작업 방식이 스스로 좋아지는 시스템"으로 확장된다.

## Evolution Loop

매뉴얼 자기진화는 다음 루프를 따른다.

1. 사용자가 새로운 매뉴얼을 등록한다.
2. 초기 매뉴얼은 단일 에이전트 작업 방식으로 실행된다.
3. [[매뉴얼-최적화-기능]]이 토큰 사용량, Verification, Time을 측정한다.
4. 반복 실행에서 병목, 검증 부족, 토큰 낭비, 불안정한 결과를 찾는다.
5. Manual이 개선안을 만든다.
6. 개선안은 사용자 승인 또는 검증 프로세스를 거쳐 매뉴얼에 반영된다.
7. 다음 실행에서 개선 효과를 다시 측정한다.

이 루프는 Manual이 한 번의 설정으로 끝나는 도구가 아니라, 실행할수록 사용자의 작업 방식에 맞게 진화하는 시스템이 되게 한다.

## Evolution Targets

자기진화 기능이 개선할 수 있는 대상은 다음과 같다.

- 단일 에이전트에게 맡기던 작업을 여러 단계로 분리한다.
- 반복되는 탐색, 정리, 변환 작업을 스크립트화한다.
- 에이전트에게 전달하기 전 입력을 전처리한다.
- 검증 단계가 없는 작업에 Verification 절차를 추가한다.
- 큰 모델이 필요 없는 단계에 더 작은 모델을 추천한다.
- 자주 실패하는 작업에 더 명확한 체크포인트를 추가한다.
- 불필요한 컨텍스트를 줄이고 필요한 근거만 전달한다.

핵심은 에이전트의 역할을 없애는 것이 아니라, 에이전트가 잘하는 판단과 추론에 토큰을 집중시키도록 매뉴얼을 바꾸는 것이다.

## Evolution Modes

자기진화는 사용자 신뢰 수준과 작업 위험도에 따라 여러 모드로 제공될 수 있다.

### Suggest Mode

Suggest Mode는 Manual이 개선안을 제안하고 사용자가 승인해야 반영하는 방식이다.

- "이 반복 작업은 스크립트화할 수 있습니다."
- "이 단계에는 추가 Verification이 필요합니다."
- "이 컨텍스트는 매번 전달하지 않고 요약본으로 대체할 수 있습니다."

MVP에서는 이 모드가 가장 안전하다. 사용자는 Manual이 어떤 이유로 매뉴얼을 바꾸려는지 이해할 수 있고, 시스템은 사용자의 도메인 판단을 유지할 수 있다.

### Assisted Edit Mode

Assisted Edit Mode는 LLM 에이전트가 개선안을 구체적인 매뉴얼 변경안으로 작성하고, 사용자는 변경 diff를 검토하는 방식이다.

이 모드는 Suggest Mode보다 적극적이다. 단순한 설명이 아니라 실제 워크플로우 단계, 전처리 규칙, 검증 체크리스트, 모델 라우팅 규칙 같은 구조화된 변경안을 만든다.

### Auto Improve Mode

Auto Improve Mode는 Manual이 낮은 위험도의 개선을 백그라운드에서 자동 적용하는 방식이다.

자동 개선의 대상은 결과물 의미를 바꾸지 않는 보조 작업부터 시작해야 한다.

- 파일 목록 캐싱
- 로그 필터링 스크립트 추가
- 반복되는 요약 입력 생성
- 테스트 결과 정규화
- 실행 메타데이터 수집 방식 개선

높은 위험도의 변경, 예를 들어 작업 절차 자체를 바꾸거나 검증 기준을 완화하는 변경은 자동 적용하지 않고 사용자 승인 대상으로 남겨야 한다.

## Relationship to Optimization

[[매뉴얼-최적화-기능]]은 현재 워크플로우를 측정하고 개선안을 만든다. 매뉴얼 자기진화 기능은 그 개선안을 매뉴얼 자체의 구조 변화로 연결한다.

- Measure는 무엇이 일어났는지 기록한다.
- Diagnose는 어디가 문제인지 찾는다.
- Recommend는 무엇을 바꿀지 제안한다.
- Self-Evolution은 그 제안을 매뉴얼 변경으로 반영한다.

따라서 자기진화 기능은 최적화 기능의 상위 경험이 아니라, 최적화 결과가 누적되어 제품 가치를 강화하는 진화 계층이다.

## Verification Guardrail

자기진화 기능은 [[신뢰성-부족]] 문제를 악화시키지 않도록 검증 가드레일이 필요하다. 매뉴얼이 스스로 바뀌면 잘못된 최적화가 누적될 수 있기 때문이다.

Manual은 다음 조건을 만족하는 변경만 자동 또는 반자동으로 반영해야 한다.

- 변경 전후의 Token Usage, Verification, Time을 비교할 수 있다.
- 검증률을 낮추는 변경은 사용자 승인 없이는 적용하지 않는다.
- 결과 의미에 영향을 줄 수 있는 변경은 사람이 검토한다.
- 자동 변경은 언제든 되돌릴 수 있는 기록을 남긴다.
- 개선 효과가 확인되지 않으면 이전 매뉴얼로 되돌릴 수 있다.

이 가드레일은 자기진화가 단순한 자동 수정이 아니라 검증 가능한 개선 루프가 되도록 만든다.

## MVP Scope

MVP에서 자기진화 기능은 완전 자동화보다 제안 중심으로 시작한다.

- 사용자가 단일 에이전트 매뉴얼을 등록한다.
- Manual이 여러 실행의 토큰, 검증, 시간 데이터를 수집한다.
- 반복되는 전처리 후보와 검증 누락 지점을 찾는다.
- LLM 에이전트가 매뉴얼 개선안을 작성한다.
- 사용자가 승인하면 매뉴얼에 반영한다.
- 다음 실행에서 개선 전후를 비교한다.

자동 개선은 낮은 위험도의 전처리, 캐싱, 요약 생성, 로그 정리 같은 보조 작업에 한정한다.
