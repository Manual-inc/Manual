# 실행 안전성 부족

실행 안전성 부족은 AI 에이전트나 자동화 스크립트가 로컬 파일, 명령어, 네트워크에 접근할 때 사용자가 충분히 안심하기 어려운 문제다.

이 문제는 [[신뢰성-부족]]과 연결되지만, 결과물의 품질보다 실행 환경의 안전성에 초점을 둔다. 에이전트가 어떤 커맨드를 실행할 수 있는지, 어떤 파일을 읽거나 쓸 수 있는지, 어떤 네트워크 호스트에 접근할 수 있는지가 명확히 제한되지 않으면 사용자는 매 실행마다 위험을 판단해야 한다.

## Problem

현재 많은 에이전트 실행 환경은 위험한 작업 앞에서 사용자 승인을 요구한다. 승인 절차는 필요한 안전장치지만, 실제 업무에서는 다음 문제가 생긴다.

- 에이전트가 커맨드를 실행할 때마다 사용자가 반복적으로 승인해야 한다.
- 복잡한 커맨드는 사용자가 위험성을 즉시 이해하기 어렵다.
- 승인했더라도 커맨드가 실제로 어떤 파일과 네트워크에 접근할지 예측하기 어렵다.
- 실수로 잘못된 커맨드가 실행되면 파일 손상, 민감 정보 노출, 외부 네트워크 접근 같은 위험이 생긴다.
- 승인 피로가 쌓이면 사용자가 내용을 충분히 검토하지 않고 승인할 가능성이 높아진다.

따라서 단순히 실행 직전에 "허용할 것인가"를 묻는 방식만으로는 안정성과 사용성을 동시에 확보하기 어렵다.

## Why Approval Alone Is Not Enough

승인은 사용자가 위험을 이해하고 판단할 수 있을 때 효과적이다. 하지만 에이전트가 생성한 커맨드는 길고 복잡할 수 있고, 셸 확장, 스크립트 호출, 도구 실행, 네트워크 요청이 함께 섞일 수 있다.

사용자는 "이 커맨드를 허용할 것인가"보다 "이 작업은 어디까지 접근해도 되는가"를 먼저 정의할 수 있어야 한다. 실행 정책이 사전에 정의되어 있으면, 에이전트나 스크립트가 실수로 다른 경로에 접근하거나 허용되지 않은 네트워크로 요청을 보내더라도 정책이 실행을 차단할 수 있다.

## User Pain

사용자가 체감하는 가장 큰 불편은 승인 피로다. 매번 승인 버튼을 누르는 것은 흐름을 끊고, 복잡한 커맨드를 해석해야 한다는 부담을 만든다.

기업이나 팀 환경에서는 문제가 더 커진다. 개인 사용자는 위험을 감수하고 승인할 수 있지만, 조직은 민감한 코드, 내부 문서, 네트워크 자원, 배포 권한을 다루기 때문에 사전 정책과 감사 가능한 실행 경계가 필요하다.

## Related System Direction

[[샌드박스-기능]]은 이 문제를 해결하기 위한 Manual의 실행 안전성 기능이다. 사용자는 파일 읽기, 쓰기, 실행 권한과 접근 가능한 네트워크 호스트를 미리 정의하고, 에이전트나 스크립트를 해당 샌드박스 안에서 실행할 수 있다.

[[LLM-Harness]]는 이런 샌드박스 정책을 실제 실행 환경에 적용하는 기반 계층이 될 수 있다. [[매뉴얼-시스템]]은 워크플로우 노드마다 어떤 샌드박스를 요구할지 정의해 실행 안정성을 작업 절차 안에 포함한다.
