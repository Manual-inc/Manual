# Manual 제품 방향 회의 원자료

이 파일은 2026-05-11 대화형 회의록을 의미 단위로 정리한 원자료다. 원문은 음성 인식 잡음이 많아 문장을 정규화했지만, 논의의 핵심 흐름과 표현은 최대한 보존했다.

## 프로젝트 개요

- 프로젝트 이름은 `Manual`.
- 모토는 "실행 가능한 매뉴얼".
- 노드를 조합해 워크플로우를 만들고, 각 노드는 스크립트나 에이전트 실행 단위가 된다.
- 에이전트는 Claude, Codex, Pi, Hermes/Homecode 계열 등 여러 종류를 등록할 수 있다.
- 데모 예시는 코드 리뷰 워크플로우다.
  - 리뷰 컨텍스트 노드가 Git URL이나 일반 URL 같은 입력을 받는다.
  - 스크립트 노드가 저장소를 clone하고 작업 디렉터리로 이동한다.
  - 테스트 코드나 정적 테스트를 먼저 실행한다.
  - 결과 리포트가 나오면 여러 에이전트가 병렬로 리뷰를 시작한다.
  - 이후 다른 에이전트들이 더블 체크한다.
  - 마지막에 결과를 모아 요약하고 코멘트를 단다.

## 현재 동작과 데모 범위

- 현재는 거의 데모 수준이지만 에이전트 실행은 돌아간다.
- Codex CLI나 Claude CLI를 로컬에서 비대화형으로 호출하고 결과 스트림을 받아 처리한다.
- 로컬 MVP라 클라우드 실행은 당장 필요하지 않다.
- 앱 서버가 있고, 여러 클라이언트가 그 서버에 JSON을 주고받는다.
- 아키텍처는 Codex 앱/CLI/웹이 서버를 공유하는 방식과 유사하게 설명됐다.
- 이 프로젝트 자체의 앱 서버를 사용하며, MCP는 현재 쓰지 않는다.
- JSON API가 더 단순하고, 필요하면 나중에 MCP로 바꿀 수 있다.
- CLI가 앱 서버와 통신해 워크플로우를 생성하고 실행할 수 있다.
- macOS UI는 생성된 워크플로우와 실행 로그를 모니터링하는 수준이다.
- 에러가 나면 노드가 에러 상태로 표시되고 빨간색 등으로 표현할 수 있다.
- CLI에는 노드 수정 기능이 일부 있지만, 현재 UI는 뷰와 모니터링에 집중되어 있다.

## 핵심 제품 방향

- 단순히 CLI가 생성한 워크플로우를 UI에서 보는 것을 넘어, 워크플로우를 단계별로 이해하고 통제할 수 있어야 한다.
- 전체 자동화도 중요하지만, 사용자가 중간에 멈추고 확인하고 다시 넘기는 흐름도 중요하다.
- 비용 절감만이 목적은 아니며, 자동화와 통제 가능성이 함께 중요하다.
- 그래도 비싼 모델에 모든 단계를 맡기는 대신, 간단한 전처리나 분석은 스크립트 또는 저렴한 로컬 모델에 맡기면 총 비용을 줄일 수 있다.
- 예시:
  - 단일 고급 모델에 1번부터 5번까지 모두 맡기면 총 100의 비용이 든다.
  - Manual에서는 간단한 단계는 5 또는 10의 비용으로 처리하고, 중요한 단계만 25 정도의 고급 모델에 맡길 수 있다.
  - 총합은 단일 고급 모델 전체 처리보다 낮아질 수 있다.

## 우선순위 기능 후보

### 부분 실행과 재시작

- 각 노드 위에 플레이 버튼이 있어야 한다.
- 마우스 오버 시 노드별 실행 버튼을 보여주는 방식이 언급됐다.
- 특정 노드부터 실행하거나, 실패한 지점에서 다시 시작할 수 있어야 한다.
- 중간 결과를 확인하고 다음 단계로 넘길 수 있어야 한다.
- 전체 플로우가 자동으로 계속 도는 것과, 한 단계씩 확인하며 진행하는 것을 모두 지원해야 한다.
- 실행 중 이상한 삽질을 하고 있으면 멈출 수 있어야 한다.
- 이전 단계의 전처리가 잘못됐는데 뒤 단계가 계속 실행되는 상황을 줄여야 한다.

### 중간 입력 주입

- 중간 지점부터 실행할 때 해당 노드의 입력값을 임의로 넣을 수 있어야 한다.
- 프론트엔드에서 컴포넌트만 독립적으로 띄우고 props/parameters를 넣어 테스트하는 Storybook과 유사한 흐름이 언급됐다.

### 노드 Storybook과 노드 관리

- 노드 하나하나를 Storybook처럼 독립적으로 보여주고 테스트할 수 있으면 좋다.
- 임의 입력을 넣고 내부 프로세스와 결과를 확인할 수 있어야 한다.
- 노드 관리 UI 또는 노드 registry/schema가 필요하다.
- 워크플로우는 등록된 노드들을 조합하는 방식이 자연스럽다.

### 토큰/비용 관측

- 워크플로우 실행마다 job/run이 남는다.
- 각 run에 전체 토큰 사용량을 저장하면 좋다.
- 노드별 토큰 사용량과 전체 비용을 볼 수 있으면 워크플로우를 개선할 때 효과를 확인할 수 있다.
- "이 워크플로우를 어떻게 고치면 토큰 비용을 덜 쓸 수 있는지 분석하고 적용" 같은 에이전트 루프도 가능하다.

### 에이전트 스킬 지정

- 에이전트 노드에 특정 skill을 지정할 수 있으면 좋다.
- 예를 들어 "이 코드 리뷰 에이전트는 code review skill을 써서 리뷰하라" 같은 식이다.
- 복수 skill 지정도 가능하다.
- 에이전트가 올바른 skill을 쓰고 있는지 보고 싶다는 요구가 있었다.
- skill을 무시하거나 다른 skill을 쓰는 상황을 감지/제어하는 방향도 언급됐다.

### 샌드박스

- Codex 코드에서 실행 시 접근 가능한 파일, 네트워크, host 등을 제한하는 기능이 있음을 확인했다.
- macOS, Windows, Linux의 네이티브 기능을 활용하는 형태로 보인다.
- 예를 들어 특정 디렉터리만 보이게 하거나, 특정 host만 조회 가능하게 할 수 있다.
- 코드 리뷰 노드는 CVE 사이트 같은 필요한 host만 접근하도록 제한할 수 있다.
- 중요하지만 발표/설명용 MVP에서는 고급 옵션으로 둘 수 있다는 의견이 있었다.

## 차별화와 포지셔닝

- 범용 워크플로우 도구와의 차이점은 "에이전트 전용"에 있다.
- 일반 LLM API 호출 노드가 아니라, Codex 같은 에이전트가 실제 저장소에서 작업하고 스킬을 사용하며 로컬 CLI와 상호작용하는 구조가 핵심이다.
- Harness-style divide/execute/evaluate loop도 Manual 워크플로우로 표현할 수 있다.
- GitHub 이슈나 다른 입력을 받아 나누는 에이전트, 실행 에이전트, 평가 에이전트, 결과 취합 에이전트로 구성할 수 있다.

## UI와 클라이언트

- macOS Swift 앱이 현재 주요 데모 클라이언트다.
- zoom in/out, 드래그 연결 같은 고급 그래프 편집 기능은 아직 없다.
- Swift로 그래프 편집을 구현하는 부담이 커서 당장은 뷰/모니터링 중심이다.
- Windows, macOS, Linux 테스트는 GitHub Actions로 돌릴 수 있다.
- 모바일은 아직 고려하지 않았고, 우선 Swift/macOS와 CLI 중심으로 보인다.

## 사업/운영 맥락

- 저장소는 현재 외부 GitHub에 있다.
- 회사 내부로 옮길지 고민이 있다.
- 아이디어를 내부에서 가져갈 수 있다는 걱정도 언급됐다.
- 잘 되면 솔루션으로 팔 수 있다는 농담 섞인 이야기가 있었다.
- 라이선스는 Apache 2.0으로 상업적 이용에 문제가 없다는 점이 언급됐다.

## 합의된 다음 범위

- 크게 욕심내지 말고 우선 다음 세 가지를 잡는다.
  - 부분 실행/재시작.
  - 노드 Storybook 또는 노드 단위 테스트/관리.
  - 토큰/비용 관측.
- 샌드박스는 중요하지만 설명용 옵션 또는 후속 작업으로 둔다.
- 에이전트 skill 지정은 방향성에 포함하되, 당장 핵심 MVP 범위와 분리해 볼 수 있다.
