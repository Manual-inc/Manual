# 매뉴얼 위키 스키마

This document defines how Manual project documentation is maintained under `docs/` using the `llm-wiki` skill structure.

## What this is

The `docs/` directory is maintained as a persistent, LLM-managed project wiki for Manual, an AI agent work system focused on two core problems:

- Token scarcity
- Reliability deficit

The human curates sources and directs project intent. The LLM maintains the structured wiki, cross-references concepts, records operations, and keeps the synthesis current.

## Directory structure

- `docs/raw/` — Immutable source documents. Do not modify files here after ingestion.
- `docs/usecase/` — Gherkin behavior contracts linked back to the wiki with `# wiki:` comments.
- `docs/wiki/` — LLM-maintained pages.
  - `sources/` — One summary page per ingested source
  - `problems/` — Problem pages that describe concrete pain points and failure modes
  - `concepts/` — Core ideas, mechanisms, and design concepts
  - `systems/` — System components, harnesses, workflows, and architectural elements
  - `features/` — Product features and capability pages
  - `architecture/` — Concrete application and execution architecture pages
  - `decisions/` — Durable product or technical decisions
  - `meetings/` — Meeting notes and normalized discussion records
  - `analyses/` — Comparisons, syntheses, and durable answers to project questions
  - [목차](목차.md) — Catalog of all wiki pages. Read this first when answering project questions.
  - `작업-로그.md` — Append-only chronological operation log.
  - `개요.md` — High-level synthesis of the wiki.

## Page conventions

- Every wiki page should have YAML frontmatter with `title`, `type`, `tags`, `sources`, `date_created`, and `date_updated`.
- Use Obsidian-compatible bracketed page names for cross-references between wiki pages.
- Cite source summaries with markdown links to `docs/wiki/sources/`.
- Do not silently overwrite contradictory claims. Add the new claim, cite it, and flag the contradiction explicitly.
- Keep source documents in `docs/raw/` immutable. If source content needs correction, add a new source or record a correction in the wiki.

## Code documentation convention

New code must be connected to the wiki. When adding code at the repository root or in source directories, include a concise comment that links to the relevant `docs/wiki/` page explaining why that code exists.

The goal is to bring code into the same maintenance loop as documentation: when `llm-wiki` updates or checks project knowledge, the domain reasoning behind code remains discoverable, cross-referenced, and easier to verify.

See [[문서-연결-코드]].

## Usecase convention

User-facing behavior specifications live under `docs/usecase/` as Gherkin `.feature` files.

- Use one `.feature` file per major system or feature.
- Write feature files in Korean and include `# language: ko`.
- Add one or more `# wiki: docs/wiki/...` comments near the top of each `.feature` file linking to the relevant wiki pages that define the behavior.
- Keep `.feature` files implementation-free. They describe user behavior, acceptance criteria, and policy expectations, not production code.
- Write `.feature` files as shared behavior contracts for CLI, API, and UI. Avoid channel-specific wording such as buttons, screens, clicks, HTTP status codes, or terminal commands unless a scenario is explicitly scoped to that channel.
- Prefer channel-neutral verbs such as "요청한다", "조회한다", "제공한다", "기록한다", "저장한다", "차단한다", and "확인할 수 있다".
- For the MVP, assume Manual is a local single-user program. Do not add authentication, authorization, role, team, or organization scenarios unless the MVP scope changes.
- Prefer scenarios that connect to wiki system pages such as [[매뉴얼-최적화-기능]], [[매뉴얼-자기진화-기능]], and [[샌드박스-기능]].
- When a system page changes materially, update the matching `.feature` file so use cases and wiki documentation remain aligned.
- Do not place generated test code or step definitions in `docs/usecase/`; implementation belongs elsewhere after a separate design decision.

## Ingest workflow

1. Read the source completely.
2. Identify the source's key claims, problem statements, entities, concepts, and system implications.
3. Create or update a summary page in `docs/wiki/sources/`.
4. Create or update relevant pages in `docs/wiki/problems/`, `docs/wiki/concepts/`, and `docs/wiki/systems/`.
5. Update `docs/wiki/개요.md` when the source changes the big picture.
6. Update `docs/wiki/목차.md` so it remains the primary navigation surface.
7. Append an operation entry to `docs/wiki/작업-로그.md`.

## Query workflow

1. Read `docs/wiki/목차.md` first.
2. Read the relevant pages.
3. Answer with links to wiki pages and source summaries.
4. If the answer becomes reusable project knowledge, offer to file it under `docs/wiki/analyses/`.

## Lint checklist

- Contradictions between problem, concept, and system pages
- Claims superseded by newer sources
- Orphan pages with no inbound links
- Frequently mentioned terms that lack their own page
- Missing cross-references between related pages
- Open questions that should be investigated before implementation
- Code comments that link to missing, stale, or unrelated wiki pages
- `.feature` files under `docs/usecase/` that no longer match their corresponding wiki system pages
- Broken or missing `# wiki: docs/wiki/...` comments in `docs/usecase/*.feature`

## Executable documentation tests

- Rust-based document rule tests live in `docs/test/`.
- The first test checks orphan documents across `docs/raw/**/*.md` and `docs/wiki/**/*.md`.
- The test also checks that every `docs/usecase/*.feature` file has at least one valid `# wiki: docs/wiki/...` comment.
- Test code must link back to the relevant wiki page that explains why the test exists. See [[문서-규칙-테스트]].

## Domain-specific notes

- "Token scarcity" means both limited context capacity and the economic cost of high-quality model usage.
- "Reliability deficit" means the absence of system-level guarantees around probabilistic model behavior, agent procedure following, verification, and repeatability.
- "LLM harness" means the foundation for handling LLM execution: tools, environment, permissions, context injection, execution control, model calls, and token budgets.
- "Manual System" means the broader work system that uses an LLM harness and other mechanisms to improve token efficiency, verification, traceability, and reliability.
- Prefer separating raw problem claims from proposed solutions. Problem pages should say what fails and why. System pages should say what mechanism may address it.
- Older root-level `wiki/` pages have been integrated into `docs/wiki/`. Treat `docs/` as the canonical wiki root for future work.
