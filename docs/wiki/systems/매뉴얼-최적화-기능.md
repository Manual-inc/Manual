# 매뉴얼 최적화 기능

매뉴얼 최적화 기능은 [[매뉴얼-시스템]]의 MVP 중심 기능이다. 이 기능은 AI 에이전트 워크플로우가 얼마나 많은 토큰을 쓰고, 얼마나 검증 가능하며, 얼마나 오래 걸리는지를 측정한 뒤 개선 지점을 제안한다.

초기 목표는 모든 AI 작업을 자동화하는 것이 아니다. 사용자가 현재 워크플로우의 비용, 시간, 신뢰도 상태를 직관적으로 보고, 다음 실행에서 무엇을 바꾸면 좋아지는지 판단할 수 있게 하는 것이다.

## MVP Thesis

Manual의 초기 매력은 "AI가 더 똑똑하게 일한다"가 아니라 "AI 작업이 얼마나 비싸고, 느리고, 불안정한지 보이며 개선할 수 있다"에서 나온다.

따라서 MVP는 기능 범위를 두 축으로 압축한다.

- Measure: AI 워크플로우의 현재 상태를 측정한다.
- Improve: 측정 결과를 바탕으로 개선 방향을 제안한다.

## Measure

Measure는 워크플로우 실행 결과를 관찰 가능한 수치와 상태로 정리하는 기능이다.

### Token Usage

Token Usage는 [[토큰-부족]]과 [[토큰-배분]] 문제를 사용자가 직접 볼 수 있게 만든다.

- 전체 토큰 사용량
- 단계별 토큰 사용량
- 모델별 토큰 사용량과 비용
- 실행별 사용량 추이

이 기능의 핵심은 단순히 토큰 수를 표시하는 것이 아니라, 어떤 단계에서 토큰이 많이 쓰이는지 보여주는 것이다.

### Verification

Verification은 [[신뢰성-부족]]을 측정 가능한 상태로 바꾸는 기능이다.

- 요구사항 충족도
- 검증 통과율
- 누락된 검증 항목
- 남은 리스크

MVP에서는 정확도와 검증 커버리지를 별도 상위 기능으로 나누지 않고 Verification 아래에 둔다. 사용자는 "결과가 맞는가"와 "무엇이 확인됐는가"를 함께 판단해야 하기 때문이다.

### Time

Time은 AI 작업이 실제 업무 흐름에서 얼마나 빠르게 끝나는지와 어디서 지연되는지 보여준다.

- 전체 작업 시간
- 단계별 소요 시간
- 재시도 횟수
- 사람 리뷰 시간

리뷰 시간은 MVP에서 직접 측정하기 어렵다면 사용자가 입력하거나 추정치로 시작할 수 있다.

## Improve

Improve는 Measure가 만든 관찰 결과를 바탕으로 워크플로우를 더 싸고, 빠르고, 검증 가능하게 바꾸는 기능이다.

### Compare

Compare는 실행 결과를 서로 비교해 개선 여부를 확인한다.

- 최적화 전후 비교
- 모델별 비교
- 워크플로우별 비교
- 반복 실행별 비교

초기 MVP에서는 "최적화 전후 비교"가 가장 중요한 비교 방식이다. 사용자는 변경이 실제로 토큰, 검증률, 시간에 어떤 차이를 만들었는지 확인해야 한다.

### Diagnose

Diagnose는 워크플로우의 병목과 불안정 지점을 찾는다.

- 토큰 낭비 지점
- 검증 부족 지점
- 시간이 오래 걸리는 단계
- 결과가 흔들리는 작업

진단은 완벽한 자동 판단보다 "사용자가 확인할 만한 의심 지점"을 명확히 보여주는 방식으로 시작한다.

### Recommend

Recommend는 진단 결과를 바탕으로 다음 실행에서 시도할 개선안을 제안한다. 핵심 방향은 작업 전체를 에이전트에게 그대로 넘기기보다, 반복적이거나 결정론적으로 처리 가능한 부분을 스크립트나 전처리 단계로 분리해 에이전트가 더 작고 정확한 입력을 받게 하는 것이다.

- 어떤 단계를 쪼갤지
- 어떤 모델을 쓸지
- 어떤 검증을 추가할지
- 어떤 컨텍스트를 줄일지
- 어떤 반복 작업을 스크립트화할지
- 어떤 원본 입력을 전처리해 전달할지

예를 들어 파일 목록 수집, 로그 필터링, 중복 제거, 형식 변환, 변경 범위 추출, 테스트 결과 요약처럼 규칙적으로 처리할 수 있는 작업은 에이전트가 직접 긴 컨텍스트를 읽게 하기보다 사전 처리 결과로 전달할 수 있다. 이렇게 하면 에이전트는 더 적은 토큰으로 더 정확한 판단을 할 수 있고, [[토큰-배분]]도 작업의 의미 판단이 필요한 구간에 집중된다.

추천은 강제 실행보다 선택지 제안으로 시작한다. Manual은 사용자가 AI 워크플로우를 이해하고 개선하도록 돕는 도구이며, MVP 단계에서는 자동 최적화보다 설명 가능한 개선 제안이 더 중요하다.

#### Preprocessing Recommendations

Preprocessing Recommendations는 Recommend의 중요한 하위 기능이다. 이 기능은 에이전트 호출 전에 수행할 수 있는 준비 작업을 찾아낸다.

- 대량 입력에서 필요한 부분만 추출한다.
- 반복되는 탐색 절차를 스크립트로 대체한다.
- 원본 데이터를 에이전트가 읽기 쉬운 요약 형식으로 변환한다.
- 실패하거나 흔들리는 판단에 필요한 근거를 먼저 정리한다.
- 에이전트가 사용할 컨텍스트를 작업 목적에 맞게 압축한다.

이 하위 기능의 목적은 에이전트의 역할을 줄이는 것이 아니라, 에이전트가 잘하는 판단, 설계, 추론 작업에 토큰을 집중시키는 것이다.

## MVP Report Shape

매뉴얼 최적화 기능의 핵심 화면은 Optimization Report로 시작할 수 있다.

```text
Optimization Report

Token Usage      82,400 tokens
Verification     73% checked
Time             14m 20s

Main Issue
Implementation step used 61% of all tokens.

Recommendation
Preprocess file discovery with a script before calling the agent.
Pass only changed files and relevant summaries into implementation.
Expected token reduction: 25-35%.
```

이 리포트는 Measure와 Improve를 한 화면에서 연결한다. 사용자는 현재 상태, 가장 큰 문제, 다음 개선안을 동시에 볼 수 있다.

## Relationship to Manual System

[[매뉴얼-시스템]]은 AI 에이전트 작업의 신뢰성, 비용 효율성, 검증 가능성을 다루는 전체 작업 체계다. 매뉴얼 최적화 기능은 그중 MVP에서 사용자에게 가장 먼저 보여줄 수 있는 제품 표면이다.

이 기능은 [[LLM-Harness]]가 제공하는 실행 로그, 모델 호출 정보, 도구 사용 기록, 컨텍스트 정보를 활용할 수 있다. 다만 MVP 문맥에서는 하네스 전체를 완성하기보다 측정 가능한 최소 실행 기록부터 수집하는 것이 우선이다.

## Relationship to Self-Evolution

[[매뉴얼-자기진화-기능]]은 매뉴얼 최적화 기능이 만든 측정, 진단, 추천 결과를 매뉴얼 자체의 구조 변화로 연결한다. 최적화 기능이 "무엇을 바꾸면 좋을지"를 보여준다면, 자기진화 기능은 그 개선안을 사용자 승인 또는 검증 가드레일을 거쳐 실제 매뉴얼 변경으로 반영한다.

## Open Questions

- Verification의 요구사항 충족도는 사람이 채점할 것인가, 별도 평가 모델이 보조할 것인가?
- 리뷰 시간은 자동 측정할 것인가, 사용자 입력으로 시작할 것인가?
- 추천 기능은 규칙 기반으로 시작할 것인가, 실행 데이터 기반으로 시작할 것인가?
- 전처리 추천은 스크립트 템플릿까지 생성할 것인가, 아니면 추천 설명만 제공할 것인가?
- Optimization Report는 단일 작업 실행 단위로 만들 것인가, 여러 실행을 묶은 워크플로우 단위로 만들 것인가?
