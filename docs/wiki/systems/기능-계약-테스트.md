# 기능 계약 테스트

기능 계약 테스트는 `docs/usecase/*.feature`에 정의된 사용자 행동 계약을 각 앱과 엔드 구현이 충족하는지 검증하는 테스트다.

`docs/usecase/`의 `.feature` 파일은 Manual의 공통 행동 계약이다. app-server, macOS 앱, Windows 앱, Android 앱, CLI 같은 각 엔드 구현은 자신이 제공하는 기능에 대해 이 계약을 충족하는 구현 테스트를 가져야 한다.

## Rules

- `docs/usecase/*.feature`에 정의된 모든 시나리오는 해당 기능을 제공하는 각 앱 또는 엔드 구현에서 통과해야 한다.
- 각 앱의 `.feature` 구현 테스트 스크립트와 step definition은 해당 앱 또는 엔드의 디렉터리 안에 둔다.
- `manual-rs/crates/app-server/`는 백엔드 앱 구현 테스트 대상이다. app-server의 기능 계약 테스트와 관련 테스트 코드는 이 crate 내부의 테스트 위치에 둔다.
- macOS 앱은 macOS 앱 디렉터리에, Windows 앱은 Windows 앱 디렉터리에, Android 앱은 Android 앱 디렉터리에 자기 구현 테스트를 둔다.
- `docs/usecase/`에는 공통 행동 계약만 둔다. 앱별 테스트 코드, step definition, fixture, driver 코드는 `docs/usecase/`에 두지 않는다.
- 앱별 구현 테스트는 채널별 표현을 검증할 수 있지만, 검증하려는 행동은 `docs/usecase/*.feature`의 채널 중립 시나리오와 추적 가능해야 한다.
- 새로운 `.feature` 시나리오가 추가되면 해당 기능을 제공하는 앱 디렉터리의 구현 테스트도 함께 추가하거나 갱신해야 한다.
- 어떤 앱이 특정 기능을 아직 제공하지 않는다면, 그 범위 결정은 별도 문서나 이슈로 명시해야 하며 조용히 테스트에서 누락하지 않는다.

## CI

CI는 각 구현 디렉터리의 기능 계약 테스트를 실행해야 한다.

- CI는 `manual-rs/crates/app-server/`, macOS, Windows, Android, CLI 등 존재하는 각 구현 디렉터리의 `.feature` 구현 테스트를 실행한다.
- 구현 테스트가 실패하면 머지할 수 없어야 한다.
- PR은 새 기능 문서, `.feature` 시나리오, 구현 테스트가 서로 맞는지 확인할 수 있어야 한다.
- CI 결과는 어떤 `.feature` 시나리오가 어느 앱 구현에서 실패했는지 추적 가능하게 출력해야 한다.

## Relationship

기능 계약 테스트는 [[테스트-규칙]]의 행동 계약 테스트 계층이다. `docs/usecase/*.feature` 파일 자체가 관련 위키 문서와 연결되어 있는지는 [[문서-규칙-테스트]]의 유스케이스 링크 검사에서 확인한다.
